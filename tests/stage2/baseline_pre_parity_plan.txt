The six failing test cases all trace to the same two root causes
(both addressed by the parity plan):

  Root cause A:  Stage 1 codegen errors with
                 `error: unsupported List member call \`concat\``
                 when compiling stage2/src/main.fuse. This is gap
                 #1 + #3 + #5 from the T4 parity investigation
                 (missing stdlib imports + hardcoded specialization
                 ordering + match-as-expression type inference).

  Root cause B:  tests/fuse/core/integration/stdlib_foundation.fuse
                 has stale expected output. The fixture asserts
                 `name, role` after `Some(alice)`, but the current
                 compiler emits `alice` for the `m.get("name")` call
                 (which is the correct behavior for a None-aware
                 Option-returning method). Pre-existing test bug
                 documented in docs/t4-parity-investigation.md as
                 "stdlib_foundation.fuse is a pre-existing test bug
                 (incorrect expected output), not a regression."

================================================================================
NOTES ON DETERMINISM
================================================================================

This baseline was captured BEFORE Wave B1 (the determinism wave).
At this commit, BuildSession.modules is still a HashMap, so repeated
runs of `cargo test stage2_core_suite` may surface different first
errors run-to-run. The "Root cause A" errors above are the ones
observed on this run; on a different run a different (but equally
real) error may surface. This is acceptable for the baseline because
all such errors trace back to the same set of compiler gaps already
documented in docs/t4-parity-investigation.md.

The B0.3 determinism harness (added in the same wave as this file)
will lock down byte-for-byte reproducibility from B1 onward. After
B1, this baseline can be regenerated and will be byte-identical
across reruns from the same commit.
